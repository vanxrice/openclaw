# TCK-003: Dynamic Model Routing (Gemma 4 31b -> 26b)

## Background & Motivation

The 31-billion parameter Gemma 4 model produces high-quality output but causes TTFT (Time To First Token) timeouts in the gateway due to massive disk load times and slow generation. This makes it challenging for a real-time system. However, its sophisticated language is ideal for Seraphina's "first contact" atmospheric greeting. We want to route the initial "boot" message to `gemma4:31b` and all subsequent real-time messages to a lighter model, `gemma4:26b`.

## Proposed Solution

Create a local OpenClaw plugin (e.g., `extensions/dynamic-router`) that implements the `before_agent_start` hook. This hook will inspect the conversation history. If the session has no prior assistant replies, it forces the model to `gemma4:31b`. Otherwise, it allows the system to fall back to the default `gemma4:26b` model.

## Implementation Steps

1. **Model Setup**: Install `gemma4:26b` via Ollama and register it in `openclaw.json` (alongside `31b`) and the `agents.defaults.models` whitelist.
2. **Set Default Model**: Change Seraphina's base model to `ollama/gemma4:26b` in `openclaw.json` for all standard real-time routing.
3. **Develop Router Plugin**:
   - Create a local plugin at `extensions/dynamic-router`.
   - Implement `openclaw.plugin.json` and `src/index.ts`.
   - Logic: Hook into `before_agent_start`. Inspect `event.messages`. If the chat history contains zero assistant replies, return `{ modelOverride: "gemma4:31b", providerOverride: "ollama" }`. Otherwise, return `{}`.
4. **Register & Enable**: Add `"dynamic-router": { "enabled": true }` to the `plugins.entries` configuration.
5. **Mitigate 31b TTFT**: Pre-load the 31b model into memory (`ollama run gemma4:31b ""`) right before sending the first message to avoid disk-load penalties triggering the gateway timeout.
6. **Validation**: Test the TTFT of the first message and verify that the second message routes to the faster 26b model via the logs.
