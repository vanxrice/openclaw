# TCK-004: Debugging First Contact Timeout

## Background

The gateway logs reveal that the "first contact" interaction is failing with a timeout (`reason=timeout provider=ollama/gemma4:31b`). When OpenClaw sends the first message to `gemma4:31b`, Ollama has to load 19GB of model weights from disk into unified memory before it can generate a single token. This "Time To First Token" (TTFT) is exceeding OpenClaw's default agent timeout window.

## Proposed Solution

We need to mitigate the TTFT timeout without altering the core routing logic (which we will handle later in TCK-003). We will achieve this by manually preloading the model into memory and investigating if OpenClaw's timeout configuration can be extended.

## Implementation Steps

1. **Model Preloading:** We will manually execute a lightweight inference directly against Ollama using the shell (`ollama run gemma4:31b "ready"`) to force the model into unified memory.
2. **Timeout Configuration (Optional):** We will investigate `openclaw.json` to see if we can extend the default timeout window for the Ollama provider to better accommodate cold starts for massive models in the future.
3. **Execution & Validation:** After preloading the model, we will attempt the "first contact" via the UI again and monitor the logs to verify that the timeout issue is resolved and the WOW moment is delivered.
